# Chroma Jack プロトタイプ実装仕様書

## 1. Codexへの指示

この仕様書に基づき、ブラウザ上で動作するゲームのプロトタイプを実装してください。

実装にあたっては、以下を厳守してください。

- React、TypeScript、MUIを使用する
- Viteでプロジェクトを構築する
- クリーンアーキテクチャを意識して責務を分離する
- ゲームロジックをReactコンポーネントから分離する
- ゲームロジックに対する単体テストを作成する
- サーバー、API、データベース、認証機能は実装しない
- ゲームは完全にブラウザ内で完結させる
- RGBの数値および16進カラーコードをプレイ中に表示しない
- 仕様書の「TODO」に記載した機能は実装しない
- 不明点があっても機能を独自に追加せず、この仕様を優先する
- 過度な抽象化は避け、プロトタイプとして読みやすいコードにする
- 実装完了後、テスト、型チェック、Lint、ビルドを実行して結果を確認する

# 2. ゲーム概要

## 2.1 仮称

**Chroma Jack**

名称は仮称であり、タイトル変更を容易にすること。

## 2.2 コンセプト

プレイヤーは、画面に表示される色だけを頼りに、現在の色を白へ近づけていく。

山札からルールで指定された枚数の色カードを公開し、その中から現在の色へ加える1枚を選ぶか、候補をすべて捨てるかを判断する。

色を加えた結果、赤・緑・青のいずれかの成分が上限を超えるとバーストとなる。

数値計算ではなく、人間の色彩感覚と直感によって判断するゲームとする。

## 2.3 プロトタイプの目的

以下を検証する。

1. 色だけを見てカードを取るか捨てるか判断する行為が楽しいか
2. 白へ近づく過程に気持ちよさがあるか
3. バーストの緊張感が成立するか
4. 数値を表示しなくてもプレイヤーが判断できるか
5. 何度も挑戦したくなるか

# 4. MVPのゲームルール

## 4.1 ゲーム全体

1ゲームは **5ラウンド** で構成する。

各ラウンドでは、ランダムな初期色から開始し、カードを加算して白に近づける。

5ラウンド終了後、各ラウンドのスコアを合計して最終スコアを表示する。

このプロトタイプは1人用のスコアアタックとする。

対人戦、オンライン対戦、CPU対戦は実装しない。

## 4.2 色の内部表現

色は内部的に以下の形式で保持する。

```ts
type RgbColor = {
  red: number;
  green: number;
  blue: number;
};
```

各値は整数とする。

通常の有効範囲は以下。

```text
0〜255
```

ラウンド進行中のプレイ画面では、以下を表示してはならない。ラウンド終了後の結果表示では、確定色またはバースト時の色のRGB値のみ表示してよい。

- RGBの各数値
- 16進カラーコード
- 残り許容量
- 白までの数値的な距離
- 加算後の予測値
- パーセント表示
- 各成分のゲージ

ラウンド終了結果では確定色またはバーストを起こした加算後のRGB値とスコアを表示してよい。それ以外のデバッグ情報は表示しない。最終結果画面ではスコアのみ表示してよい。

注入されたゲームルールが一部の色成分のバースト継続を許す場合は、許容数まで超過成分を255へ固定してラウンドを続行する。許容数は0〜2色とし、許容量を超えてラウンドが終了した結果画面には、累計で何色がバーストしたかを表示する。

ゲーム開始前にルールを選択できるようにし、初期選択はClassicとする。バーストした色成分を上限へ固定して続行する場合は、プレイ画面に累計のバースト色数を表示する。

表示文言はJSON辞書から取得し、日本語と英語を開始画面で切り替えられるようにする。ゲームルール本体は表示文言を持たず、選択可能なルール設定がルールと翻訳キーを関連付ける。

## 4.3 初期色

各ラウンド開始時に、以下の範囲から各RGB成分をランダム生成する。

```text
0〜127
```

例：

```ts
{
  red: randomInteger(0, 127),
  green: randomInteger(0, 127),
  blue: randomInteger(0, 127),
}
```

初期色が暗めになるよう範囲を制限し、カードを複数回取る余地を確保する。

## 4.4 ゲームカード

カードは表示用の色と、使用時に適用する効果を持つ。現在の通常カードは、
RGBそれぞれの加算値を `AddColorEffect` として保持する。

```ts
new GameCard({
  id,
  displayColor,
  effect: new AddColorEffect(amount),
});
```

`GameRound` が選択されたカードの効果を現在の `Hand` へ適用し、
未選択候補の破棄と次候補の公開を行う。カード種別を増やす際は、
カード種別ごとの分岐を増やすのではなく `CardEffect` の実装を追加する。

各成分は以下の範囲からランダム生成する。

```text
0〜160
```

ただし、完全な黒である次のカードは生成しない。

```ts
{
  red: 0,
  green: 0,
  blue: 0,
}
```

カードの色は、そのカードが持つRGB値をそのまま背景色として表示する。

カードのRGB数値やカラーコードは表示しない。

## 4.5 山札

各ラウンド開始時に、ルールで指定された枚数の色カードを生成する。Classicは12枚、Clamp Challengeは24枚とする。

カードは生成された順番から、ルールで指定された候補枚数を同時に公開する。Classicは1枚、Clamp Challengeは3枚とする。

山札が空になった場合、その時点でラウンド終了とする。

山札のカード一覧や、公開候補より先のカードは表示しない。

残り枚数は表示してよい。

表示例：

```text
残り 8枚
```

## 4.6 プレイヤーの操作

現在公開されている候補に対し、以下のいずれかを選択する。

### 「加える」

候補から選択した1枚のRGB各成分を現在色へ加算し、選ばなかった候補はすべて破棄する。

```ts
nextColor = {
  red: currentColor.red + card.color.red,
  green: currentColor.green + card.color.green,
  blue: currentColor.blue + card.color.blue,
};
```

加算後にバースト判定を行う。

バーストしなければ、その色を新しい現在色とし、次の候補を公開する。

### 「捨てる」

現在の色を変更せず、公開中の候補をすべて破棄する。

その後、次の候補を公開する。

捨てられる枚数に制限は設けない。

ただし、山札は有限なので、カードを捨てるほど白へ近づく機会が減る。

### 「ここで止める」

現在の色を確定し、そのラウンドを終了する。

最初のカードを操作する前でも止められるものとする。

確認ダイアログは表示しない。

## 4.7 バースト

カードを加えた結果、いずれか1つでも255を超えた場合、バーストとする。

```ts
const isBurst =
  nextColor.red > 255 || nextColor.green > 255 || nextColor.blue > 255;
```

バーストした場合、そのラウンドは即座に終了する。

バーストしたラウンドのスコアは **0点** とする。

オーバーフロー、255への切り捨て、該当成分のみの除外などは行わない。

バースト後は、短い演出またはメッセージを表示し、プレイヤーが結果を確認できるようにする。

表示例：

```text
バースト
白を通り越してしまいました
```

プレイ中の判断材料として数値は表示しない。バースト後のラウンド結果では、振り返りのために加算後のRGB値を表示する。

## 4.8 ラウンド終了条件

以下のいずれかでラウンド終了とする。

- プレイヤーが「ここで止める」を選択した
- 山札をすべて使い切った
- バーストした

## 4.9 スコア

白 `RGB(255, 255, 255)` との距離を用いてスコアを算出する。

距離はRGB空間上のユークリッド距離とする。

```ts
const distance = Math.sqrt(
  Math.pow(255 - color.red, 2) +
    Math.pow(255 - color.green, 2) +
    Math.pow(255 - color.blue, 2),
);
```

最大距離は黒と白の距離。

```ts
const maxDistance = Math.sqrt(Math.pow(255, 2) * 3);
```

ラウンドスコアは0〜1000の整数とする。

```ts
const normalized = 1 - distance / maxDistance;
const score = Math.round(normalized * 1000);
```

以下を保証する。

```ts
score = Math.max(0, Math.min(1000, score));
```

完全な白なら1000点。

完全な黒なら0点。

バースト時は色に関係なく0点。

最終スコアは5ラウンド分の合計で、0〜5000点となる。

## 4.10 同じ色の再現性

通常プレイではブラウザの乱数を使用してよい。

乱数生成処理はインターフェースとして分離し、テスト時には固定値を返す実装へ差し替えられるようにする。

## 5.1 画面構成

2. ゲーム画面
3. 最終結果画面

```ts

```

## 5.2 開始画面

表示要素：

- 仮タイトル「Chroma Jack」
- 短い説明
- 「ゲームを始める」ボタン

説明文：

```text
色を重ねて、白に近づけよう。
ただし、加えすぎるとバースト。
```

表示してはならないもの：

- 詳細なRGB計算式
- RGB値
- 色コード
- 長いチュートリアル

## 5.3 ゲーム画面

### 上部情報

以下を表示する。

- 現在ラウンド
- 全ラウンド数
- これまでに確定した合計スコア
- 山札の残り枚数

表示例：

```text
ラウンド 2 / 5
合計 842点
残り 7枚
```

現在ラウンドの途中スコアは表示しない。

### 現在色

現在色を大きな長方形または正方形として表示する。

- 画面内で最も目立つ要素にする
- 背景色として現在色を使用する
- 数値や色コードを重ねない
- ラベルとして「現在の色」のみ表示してよい

### 公開候補

現在公開されている候補を、現在色の下に選択可能なカードとして表示する。

- カードの背景色にカード色を使用する
- 数値や色コードを表示しない
- ラベルとして候補番号のみ表示してよい

現在色を上段、公開候補を下段に表示し、操作ボタンはその下へ配置する。

### 操作ボタン

以下の2ボタンを表示する。候補色そのものが加算操作を兼ねる。

- 「候補をすべて捨てる」
- 「ここで止める」

優先度は以下。

- 「加える」：contained
- 「捨てる」：outlined
- 「ここで止める」：textまたはoutlined

誤操作防止の確認ダイアログは設けない。

### 加算演出

「加える」を押した場合、現在色が新しい色へ切り替わったことが分かる簡単なCSSトランジションを付けてよい。

ただし、複雑なアニメーションライブラリは導入しない。

## 5.4 ラウンド結果表示

# 6. UI・デザイン方針

## 6.1 基本方針

色そのものを主役にする。

MUIのコンポーネントを活用しつつ、装飾を増やしすぎない。

以下を意識する。

- 余白を広く取る
- 操作箇所を分かりやすくする
- 色面を大きく見せる
- 白、黒、グレーを基本UI色とする
- カード色とUI色を競合させない
- 強いグラデーションや派手な背景を使わない
- カタカナ英語を多用しない

## 6.2 背景との境界

表示色が白または黒に近い場合でも形状を認識できるよう、色面には常に境界線を付ける。

```text
1px solid rgba(0, 0, 0, 0.2)
```

またはMUIテーマに追従する同等の境界線を使用する。

## 6.3 ボタン配置

スマートフォンでは縦並びでもよい。

デスクトップでは横並びを基本とする。

タップ領域は十分に確保する。

## 6.4 ダークモード

プロトタイプでは実装しない。

OS設定への自動追従も不要。

# 7. アーキテクチャ

## 7.1 方針

フロントエンドだけの小規模アプリであるため、クリーンアーキテクチャを過剰に形式化しない。

ただし、少なくとも以下を分離する。

- ゲームのルール
- ゲームを進行させるユースケース
- 乱数生成などの外部依存
- Reactによる表示と操作

依存方向は内側へ向ける。

```text
Presentation
    ↓
Application
    ↓
Domain

Infrastructure
    ↓
Application / Domainで定義されたインターフェース
```

Domain層はReact、MUI、ブラウザAPIに依存してはならない。

## 7.3 Domain層

Domain層には以下を含める。

### エンティティ

- RGB色
- ゲームカードとカード効果
- ラウンド
- ゲーム

色の加算とバースト判定は `Hand`、カード適用と候補進行は
`GameRound`、ゲーム全体の遷移は `GameEngine` が担う。

### 制約

- Reactをimportしない
- MUIをimportしない
- `window`や`document`を参照しない
- ランダム値を直接生成しない
- 表示用文言を持たせない

## 7.4 Application層

ゲーム操作をユースケースとして表現する。

最低限、以下の操作を実装する。

```ts
startGame();
acceptCurrentCard();
discardCurrentCard();
standCurrentRound();
startNextRound();
returnToTitle();
```

不正な状態で操作された場合は、状態を壊さないようにする。

例：

- ラウンド終了後にカードを加えない
- 山札が空なのにカードを捨てない
- 最終結果画面から次のラウンドを開始しない

プロトタイプのため、例外を画面に表示する必要はない。

## 7.5 Infrastructure層

乱数生成器を実装する。

```ts
export interface RandomGenerator {
  nextInteger(min: number, max: number): number;
}
```

`BrowserRandomGenerator`では `Math.random()` を使用してよい。

テストでは固定値または事前に指定した値の列を返す実装を使う。

## 7.6 Presentation層

Reactコンポーネントとカスタムフックを配置する。

コンポーネント内で以下を直接実装しない。

- RGBの加算
- バースト判定
- スコア計算
- 山札生成ロジック
- ラウンド終了判定

React側は、Application層の操作を呼び出して状態を描画することに専念する。

# 8. 状態モデル

実装時の参考として、ゲーム状態を以下のように定義する。

```ts
type GamePhase = 'notStarted' | 'playing' | 'roundFinished' | 'gameFinished';

type RoundEndReason = 'stood' | 'deckExhausted' | 'burst';

type RoundResult = {
  roundNumber: number;
  finalColor: RgbColor;
  score: number;
  endReason: RoundEndReason;
};

type GameState = {
  phase: GamePhase;
  currentRound: GameRound | null;
  roundResults: RoundResult[];
};

type GameRound = {
  roundNumber: number;
  hand: Hand;
  offeredCards: GameCard[];
  remainingDeck: GameCard[];
};
```

必要に応じてプロパティを追加してよいが、UI都合の状態をDomainエンティティへ混入させない。

# 9. ゲーム進行の詳細

## 9.1 ゲーム開始

「ゲームを始める」を押したとき：

1. 過去のゲーム状態を初期化する
2. ラウンド番号を1にする
3. 初期色を生成する
4. ルール指定枚数の山札を生成する
5. 山札の先頭からルール指定枚数を候補として公開する
6. ゲーム画面へ移動する

## 9.2 カードを加える

1. クリックされた公開候補1枚と現在色を即座に加算する
2. バーストを判定する
3. バーストならラウンド結果を0点で確定する
4. バーストしなければ現在色を更新する
5. 選ばなかった候補を破棄し、次の候補を公開する
6. 次の候補がなければ、山札切れとしてラウンド結果を確定する

## 9.3 カードを捨てる

1. 現在色は変更しない
2. 公開中の候補をすべて破棄する
3. 次の候補を公開する
4. 次の候補がなければ、山札切れとしてラウンド結果を確定する

## 9.4 ここで止める

1. 現在色からスコアを計算する
2. 終了理由を `stood` とする
3. ラウンド結果を確定する

## 9.5 次のラウンド

現在ラウンドが5未満の場合：

1. ラウンド番号を1増やす
2. 新しい初期色を生成する
3. 新しい山札を生成する
4. 次のラウンドを開始する

現在ラウンドが5の場合：

1. ゲーム終了状態にする
2. 最終結果画面を表示する

# 10. テスト仕様

## 10.1 Domain層

最低限、以下をテストする。

### 色加算

- 各RGB成分が正しく加算される
- 入力オブジェクトが破壊的に変更されない

### バースト判定

- 255以下ならバーストしない
- 赤が256ならバーストする
- 緑が256ならバーストする
- 青が256ならバーストする
- 複数成分が256以上でもバーストする
- ちょうど255はバーストしない

### スコア計算

- 白は1000点
- 黒は0点
- 中間色は0〜1000点
- 白に近い色ほど高得点
- 結果が整数になる
- バースト時のラウンドスコアは0点

## 10.2 Application層

最低限、以下をテストする。

- ゲーム開始時にラウンド1が作成される
- 初期色が各成分0〜127の範囲になる
- ルールで指定した枚数のカードが生成される
- ルールで指定した枚数の候補が公開される
- カードの各成分が0〜160の範囲になる
- 完全な黒のカードが生成されない
- 「加える」で現在色が更新される
- 「捨てる」で現在色が変化しない
- 「ここで止める」でラウンド終了になる
- 山札を使い切るとラウンド終了になる
- バーストすると0点になる
- 5ラウンド終了後にゲーム終了になる
- 最終スコアがラウンドスコアの合計になる

## 10.3 Presentation層

主要な操作のみテストする。

- 開始画面からゲームを開始できる
- ゲーム画面に現在色と公開候補が表示される
- ラウンド進行中はRGB値や16進カラーコードが画面に表示されない
- ラウンド終了後は確定色またはバースト時のRGB値が表示される
- 候補色をクリックして即座に加えられる
- 候補破棄と停止の2つの操作ボタンが表示される
- ラウンド終了後に次ラウンドへ進める
- 最終結果が表示される
- 「もう一度遊ぶ」で新しいゲームを開始できる

スナップショットテストは必須としない。

# 11. 品質要件

## 11.1 TypeScript

- `strict`を有効にする
- 原則として`any`を使用しない
- 非nullアサーションの多用を避ける
- ドメイン型を明確に定義する
- 型エラーを残さない

## 11.2 コード品質

- 1つの関数に複数の責務を持たせない
- UIコンポーネントを巨大化させない
- マジックナンバーを定数化する
- ゲームルールを`GameRules`へまとめ、ゲーム開始時に外から注入できるようにする

クラシックルールの例：

```ts
const rules = GameRules.classic();
const game = new GameEngine(rules, randomGenerator);
```

初期色範囲、色生成傾向、RGB成分ごとの超過処理、クランプ時のスコア上限はPolicyとして差し替え可能にする。Domain Modelからグローバル設定を直接参照せず、1ゲーム中は開始時に注入された同じルールを使用する。

## 11.3 アクセシビリティ

ゲームの性質上、色そのものを判断材料とするため、完全な色覚アクセシビリティはMVPの対象外とする。

ただし、一般的な操作性として以下は対応する。

- ボタンをキーボードで操作できる
- ボタンに明確なテキストを付ける
- 十分なタップ領域を確保する
- フォーカス表示を消さない
- 状態変化をテキストでも伝える
- 色面に適切な`aria-label`を付ける

`aria-label`にもRGB値やカラーコードは含めない。

例：

```text
現在の色
次に引いた色
```
